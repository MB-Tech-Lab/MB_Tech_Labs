# Time Tracking

## Purpose

The time tracking system captures accurate work hours and enables verification of task-based work, performance measurement, project billing, and accountability. Time tracking is mandatory and integral to performance evaluation.

---

## Working Hours

### Standard Work Day
- **Work Window**: 9 AM – 5 PM (your local time)
- **Duration**: 8 hours per day (or agreed part-time equivalent)
- **Days**: Monday – Friday
- **Breaks**: Lunch breaks not required to be logged, 15-min breaks typical

### Flexible Arrangements
- Part-time work: Different daily target (documented)
- Flexible hours: Different start/end time (approved)
- Time zone: Work hours in your time zone
- **Requirement**: Communication with Team Lead on schedule

### Non-Working Days
- Weekends: Saturday & Sunday (not counted)
- Holidays: Documented company holidays (not counted)
- Timeoff: Planned vacation/sick leave time (not counted)

## Timer System

###Start Timer
**Who**: Developer can start timer for assigned task

**Process**:
1. Open task details
2. Click "Start Timer" (or start work)
3. Confirm task relationship
4. Timer begins (visible real-time)

**What Happens**:
- Timer runs continuously until stopped
- Elapsed time displayed on screen
- Time entry created in background
- Synced to system

**Best Practice**: Start timer immediately when beginning work

### Work Session Recording
**During Work**:
- Timer shows elapsed time
- Can see current status on dashboard
- Real-time updates to team metrics
- Cannot edit time while timer running

**Monitoring**:
- System detects if computer inactive (future enhancement)
- Long continuous sessions flagged
- Idle periods detected (if enabled)

### Stop Timer
**When Work Complete or Breaking**:
1. Click "Stop Timer"
2. Timer stops
3. Duration calculated automatically
4. Time entry recorded with:
   - Start timestamp
   - End timestamp
   - Duration (minutes/hours)
   - Task associated
   - Date

**After Stopping**:
- Time entry saved
- Can view in daily summary
- Visible to Team Lead/Manager

## Manual Time Entry

### When to Use
- Timer forgot to start
- Work done without timer
- Administrative/meeting time
- Retroactive logging (within limits)

### Manual Entry Process
1. Select task or activity
2. Enter start time
3. Enter end time
4. Duration calculated automatically
5. Add notes (optional but recommended)
6. Submit for approval
7. Team Lead reviews and approves

### Manual Entry Rules
- Cannot backdate more than 7 days
- Requires Team Lead approval
- 15-25% of time entries can be manual (guidelines)
- Excessive manual entries flagged
- Must have legitimate reason

### Manual Entry Approval
- Team Lead receives notification
- Reviews entry and notes
- Approves if reasonable
- Rejects if suspicious
- Asks for clarification if needed

## Daily Time Aggregation

### End-of-Day Summary
System automatically:
- Totals all time entries for the day
- Calculates break time (if tracked separately)
- Shows daily total
- Compares to 8-hour target
- Flags variances

### Daily Time Target
- **Target**: 8 hours logged per day = 40 hours per week
- **Acceptable Range**: 7.5 - 8.5 hours (minor variance)
- **Below Target**: Flagged, explanation requested
- **Above Target**: Noted (may indicate overwork or extra tasks)
- **Part-Time**: Adjusted target documented

### Daily Review
- Developer can review own daily summary
- See list of all entries
- Verify accuracy
- Make adjustments if needed (before completion)
- Submit for finalization

### End-of-Week Report
- Weekly total hours calculated
- Daily breakdown visible
- On-time compliance tracked
- Trends analyzed
- Reported to Team Lead/Manager

## Compliance & Rules

### Mandatory Requirements
- Time log all work within system
- Time entries must be accurate and honest
- Log time daily (not bulk weekly entry)
- Start/stop times must reflect actual work
- No fabricated or inflated entries
- Must reach daily target consistently

### Prohibited Actions
- Creating fake time entries
- Manipulating timer duration (backdating more than 7 days)
- Sharing timer credentials with others
- Having someone else log time for you
- Keeping timer running while not working (idle abuse)
- Inflating hours beyond actual work

### Acceptable Practices
- Starting/stopping timer multiple times per day (as needed)
- Taking breaks without timer running (doesn't count against 8-hour target)
- Manual entry for forgotten timer start (with reason)
- Logging meeting time to "Meetings" task
- Logging non-task time to "Administration" or similar

## Idle & Inactivity Flags

### What Gets Flagged
- **Idle Session**: Timer running 15+ continuous minutes with no activity
- **Night Hours**: Timer used outside 9 AM - 5 PM window
- **Very Long Session**: Single task entry > 4 hours
- **Unusual Pattern**: Time entries at unusual times
- **Rapid Switching**: Too-frequent timer starts/stops

### Flag Triggers
- Automated system detects patterns
- Unusual entries highlighted
- Team Lead can manually flag entry
- Admin can monitor trends

### Management Response
- Flag reviewed by Team Lead
- Explanation requested from developer
- May ask for additional detail
- Legitimate reasons accepted
- Suspicious patterns escalated

### Potential Outcomes
- **Legitimate**: Accepted and recorded
- **Clarified**: Documented explanation added
- **Adjusted**: Time entry corrected
- **Rejected**: Entry marked invalid, removed from records
- **Severe**: Policy violation investigation

## Time Tracking Scenarios

### Scenario 1: Normal 8-Hour Day
- 9:00 AM - 10:30 AM: Task A (1.5 hours)
- 10:30 AM - 12:00 PM: Task B (1.5 hours, includes break)
- 12:00 PM - 1:00 PM: Lunch (not logged)
- 1:00 PM - 3:30 PM: Task C (2.5 hours)
- 3:30 PM - 5:00 PM: Task D (1.5 hours, includes break)
- **Total**: 7.5 hours (acceptable)

### Scenario 2: Longer Day with Overwork
- 9:00 AM - 12:00 PM: Task A (3 hours)
- 12:00 PM - 1:00 PM: Lunch
- 1:00 PM - 6:00 PM: Task B (5 hours, important deadline)
- **Total**: 8 hours (acceptable, though 5-hour session flagged)
- **Note**: Was this overtime, or task underestimated?

### Scenario 3: Partial Day
- 9:00 AM - 12:00 PM: Task A (3 hours)
- 12:00 PM - 1:30 PM: Lunch + appointment
- 1:30 PM - 5:00 PM: Task B (3.5 hours)
- **Total**: 6.5 hours (below target, explanation requested)

### Scenario 4: Manual Entry Needed
- 9:00 AM - 12:00 PM: Timer used (3 hours)
- 12:00 PM - 2:00 PM: Meeting (no timer)
- 2:00 PM - 5:00 PM: Forgot to start timer, manual entry (3 hours)
- **Manual Entry**: 2:00 PM - 5:00 PM, Task B
- **Total**: 9 hours (overwork but reasonable given meeting)

## Best Practices

### For Developers
- Start timer immediately when beginning work
- Stop timer when taking breaks
- Use accurate times (don't round)
- Log time daily (not bulk on Friday)
- Reach 8-hour target consistently
- Use manual entry only when necessary
- Ask Team Lead if unsure

### For Team Leads
- Monitor time patterns weekly
- Flag suspicious entries promptly
- Ask for clarification respectfully
- Approve legitimate entries quickly
- Reject fabricated entries
- Track patterns for coaching
- Report concerns to Project Manager

### For Project Managers
- Use time data for project estimation
- Identify efficiency trends
- Allocate time budget wisely
- Flag repeated issues
- Use data for team/individual feedback

## Timer Workflow

### Start Timer
- User selects an assigned task
- Confirms task details in the system
- Clicks "Start Timer"
- Timer begins tracking the work session

### Work Session Recording
- Timer continues running while user works
- User can view elapsed time
- Real-time synchronization to system

### Stop Timer
- User completes work on task
- Clicks "Stop Timer"
- Duration is calculated and recorded
- Total daily hours updated

## Logged Data Per Time Entry
- **Task ID**: Associated task
- **Start Timestamp**: When work began
- **End Timestamp**: When work ended
- **Duration**: Total time logged
- **Date**: Calendar date
- **Status**: Active, Paused, or Completed
- **User**: Who logged the time
- **Project**: Associated project for billing/reporting
- **Billable Flag**: Yes/No for client billing
- **Notes**: Optional context or work description

## Daily Time Aggregation
- System totals all time entries per user per day
- Total should generally equal 8 hours (less breaks)
- Partial days are tracked and reported
- Non-working days (weekends, holidays) are excluded

## Manual Time Entry
- Users may manually log time if timer wasn't used
- Requires:
  - Task selection
  - Start and end times
  - Duration calculation
  - Justification or context
- Subject to Team Lead or Manager approval
- Creates audit trail for review

## Idle and Inactive Time
The system may track:
- Gaps between timer sessions during work hours
- Time logged to a task but no work action recorded (based on future activity tracking)
- Extended periods without timer activity

**Admin/Manager Review**:
- Idle sessions are flagged for follow-up
- Context is requested from user if needed
- Patterns of idle time affect performance ratings

## Time Tracking by Role

### Intern / Employee
- Can start/stop timer for assigned tasks
- Can view own daily time logs
- Can submit manual time entries with approval
- Cannot edit or delete past entries

### Team Lead
- Can view team member time logs
- Can flag questionable entries for review
- Can request clarification on time entries
- Can override time entries if necessary (with audit note)

### Manager
- Can view team member time logs for assigned teams
- Can analyze time patterns and efficiency
- Can identify resource gaps or concerns
- Can generate time reports for Admin submission

### Admin
- Can view all user time logs across entire system
- Can override time entries with justification
- Can generate system-wide time analytics
- Can audit time data for compliance

## Compliance Rules
- Users must time log all work within the system
- Time entries should be accurate and honest
- Fake or inflated entries are policy violations
- Timers should reflect actual task work
- Gaps without explanation may affect performance

## Billing and Allocation
- Billable flag indicates time to bill to clients
- Non-billable time (training, admin, etc.) is tracked separately
- Projects track billable vs. non-billable hours
- Manager submits billable time for invoicing
- Cost analysis uses time data for project profitability
